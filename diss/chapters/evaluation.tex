\section{Functional Evaluation}
\label{sec:4.1}
Throughout implementation I had been iteratively adjusting the FastMDP algorithm and reward functions to drive desirable behaviour. As described in the Implementation chapter this process was driven partly by qualitative analysis of the flight paths and partly by manual simulated annealing of my reward function parameters. 
After completing the project core, I wanted to more strictly examine the pursuit and evasion behaviour of the agents to inform the success of the project.

I designed a set of 1v1 validation scenarios, each defined by a specific combination of initial values and agent parameter ratios, ranging in complexity. Each scenario was run for up to 1000 timesteps or until a capture, and at each timestep the agent values were recorded and checked against predefined validation criteria. A scenario is considered valid if all tracked values satisfy these criteria throughout the run. The exact values of the scenarios and their validation results can be found in Appendix C. 

\subsection{Validity Definitions}
\label{sec:4.1.1}
I defined two types of validity and the criteria for them between timesteps.

\textbf{Kinematic Validity}

I established a set of three kinematic parameters and their realistic ranges.

\begin{itemize}[nosep]
    \item Speed $s \in [34.3, 103.0]$ m/s (0.1 - 0.3 Mach)
    \item Azimuth rate $\dot{\psi} \in [-1.3, 1.3]$ rad/s
    \item Altitude $z > 0$ m
\end{itemize}

A scenario is considered kinematically valid if these hold for all agents at all timesteps.

\textbf{Behavioural Validity}

Let $d(t) = ||\mathbf{p}_1(t) - \mathbf{p}_2(t)||$ be the Euclidean distance between a pair of agents with position functions $\mathbf{p}_1$ and $\mathbf{p}_2$. \\
Further let $d_0 = d(0)$ the initial separation and $d_{\min} = \min_{t \in [0, 1000]} d(t)$ the minimum separation of the run.

A scenario is considered behaviourally valid if:

\begin{enumerate}[nosep]
    \item $\exists\, t.d(t) < d(t-1)$ --- i.e. $d(t)$ does not monotonically increase throughout the run.
    \item $d_{\min} < d_0$ OR capture occurs during the run.
\end{enumerate}

Together these ensure that the reward functions produce meaningful interaction -- a non-increasing $d(t)$ at some point implies at least one agent is closing on the other's position, and $d_{\min} < d_0$ confirms this response results in net closure over the course of the run.


If all agents are kinematically valid, and all pairs of agents are behaviourally valid, then the scenario is valid.

\subsection{Results Summary}
\label{sec:4.1.3}

All scenarios passed the behavioural validity criteria, and all but one passed kinematic validity (with scenario 7 failing the hard-deck constraint as described in \ref{sec:4.1.2}). The implemented reward functions and parameter tuning drive acceptable pursuit-evasion behaviour, but closer examination of key parameters (speed, turn rates, separation) revealed limitations with the positive and negative rewards.

Since the reward functions are mostly alignment-based, turning artefacts are observed especially at close ranges. The positive reward

\begin{equation}
    R^+ = \frac{\hat{v}_{self} \cdot \hat{r} + \hat{r} \cdot \hat{v}_{enemy}}{d}
\end{equation}
\label{eq:4.1}

is maximised when the agent points towards the opponent and the opponent points away. The negative reward

\begin{equation}
    R^- = \frac{-\hat{r} \cdot \hat{v}_{enemy} + \hat{r} \cdot \hat{v}_{enemy}}{d}
\end{equation}
\label{eq:4.2}

penalises the inverse configuration. They both scale with $1/d$, so at close ranges small changes in velocity direction produce large changes in reward, incentivising sharp movements rather than smooth adjustments. Furthermore, agents can increase their positive rewards by adjusting their own alignments relative to their opponent, producing the bouncing behaviour observed in scenarios 3 and 6.

The hard-deck penalty did not prevent altitude violations, possibly due to the forward projection horizon not considering states near $z = 0$ and not encountering the penalty during planning.

\subsection{Testing Strategy}
\label{sec:4.5.1}

The GitHub actions CI testing pipeline was robust and ran automatically on every remote commit. Using generative AI allowed for easy iteration of tests to match system refactors, but this process was implementation-led rather than test-driven.

Coverage suffered during periods of heavy refactoring, especially when trying to introduce optimisation libraries including \texttt{numpy} and \texttt{numba}. These changed the core structure of the code, and invalidated existing tests because they no longer matched interfaces or mocked objects. Further consideration could have been taken when designing module interfaces, or planning the progression from 1D to 3D simulation to ensure that a test suite could evolve alongside the codebase and provide more value. \\
The typical workflow was to implement a component or refactor an existing module, verify it through manual program runs, and then generate tests to confirm its behaviour. Tests became indicators of previous correctness rather than the basis for refactors and new functionality.

The pipeline did catch actual issues - for example, a failing run on commit \texttt{68e85c6} identified that the output manager did not support the newly-integrated pyvista visualisation outputs, but overall a key lesson learnt was to take a more test-driven methodology.

\subsection{Agile Effectiveness}
\label{sec:4.5.2}

Three-week agile sprints guided the development schedule with short reviews of completed functionality written after each sprint. The core benefit was iterative expansion, where it helped me stay on track to implement base functionality whilst reacting to changing requirements in the codebase. An example of this was the \texttt{numba} library integration, where I dedicated time to adjust the central simulation framework to procedural methods before reviewing the work and continuing with other features.

Before beginning a sprint I used the previous review to guide functionality for the next one, but was flexible in considering different modules or components that made the most sense to implement next. The reviews show a clear progression of design decisions towards the success criteria of the project.

To further improve the effectiveness of the methodology I could have incorporated a ticketing system to track the implementation progress of more granular features, but as a solo developer with a comprehensive design plan for the codebase I felt confident with the methodology and its flexibility.
